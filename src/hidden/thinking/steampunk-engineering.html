Steampunk Engineering Philosophy

Building resilient AI systems where every lever has a purpose you understand, every whistle tells you something specific, and when something breaks, you know which bolt to tighten.

The Core Insight: Agency Distinction

The key to building sustainable AI-assisted systems lies in distinguishing between two types of agency: strategic agency (what and why) that humans must retain, and tactical agency (how) that can safely be automated.

Key Principle: Automate the "how," preserve human control over the "what" and "why."

Strategic vs Tactical Agency

Strategic Agency (Human-Controlled)

  • Define what "optimal" means for your context
  • Decide what problems to solve
  • Choose what to build and why
  • Set success criteria and review processes

Tactical Agency (Automation-Friendly)

  • How to implement solutions efficiently
  • Repetitive execution patterns
  • Boilerplate and setup tasks
  • Following established patterns

Steampunk vs Cybersuit Engineering

The Steampunk Approach

  • Every lever has a purpose you understand
  • Every whistle tells you something specific

The Cybersuit Problem

  • Black box optimization you can't debug
  • "It just works" until it doesn't

Component Replaceability Principle

"I just like my shit to break into component parts that can be replaced. I hate magic, especially when it breaks."

The Debugging Test

When something breaks, can you:

  1. See what it's doing? (logs, files, clear state)
  2. Test each piece separately? (modular components)
  3. Replace the broken part? (swappable tools)
  4. Understand the failure mode? (no black boxes)

If you can't answer yes to all four questions, your system has too much magic. Search broken? Swap it out. Want better embeddings? Drop in a replacement. Context system acting up? Replace the whole thing. This only works when components have clear boundaries.

Software Winter Survival Strategy

The Boat Test

Your tools should work in extreme conditions where connectivity is limited or nonexistent:

  • In the desert: Solar panel + laptop + local tools = productivity
  • On a boat: Starlink goes down? Local AI keeps working
  • In space: No ground communication? Semantic search still finds what you need
  • Under the sea: Submarine internet sucks? Ollama doesn't care

AI as Computation, Not Service

The Reality Check:

AI = Matrix multiplication + nonlinear functions + lots of parameters = pattern matching

Once you download the weights, it's your computation. No different from having a compiler or database on your machine.

The 100 Rabbits Principles Applied to AI

From their software winter survival guide, principles that translate directly to AI tooling:

  • Immediate: Tools work right now, not after setup/login/payment
  • Repairable: When something breaks, you can fix it yourself
  • Portable: Runs anywhere you can run code
  • Durable: Will work in 10 years without updates
  • Efficient: Doesn't need massive resources to be useful

Implementation

  • Local ollama models: No internet required
  • File-based everything: No databases to maintain
  • Bash/Python tooling: Runs on any Unix system
  • Self-contained search: Your embeddings, your index
  • Human-readable storage: Debug with cat and grep

Optimization Pressure Safeguards

"Optimality" acts as selective pressure that shapes agents, potentially leading to misalignment or loss of human agency. The steampunk approach provides natural safeguards:

  • Human-defined objectives: You set what gets optimized
  • Controlled feedback loops: You control what data the system learns from
  • Maintained veto power: Human review is integral, not optional
  • Preserved exit strategies: Rollbacks, alternatives, documentation
  • White box optimization: You define and control objectives

The Pattern

Good

Human → AI → Information → Human Decision → Automated Execution

Bad

Human → AI → Decision → Action

Why This Approach Wins Long-Term

Resilience Factors

  1. Reliability: Mechanical systems fail predictably
  2. Maintainability: You can fix what you can understand
  3. Portability: Tools run anywhere, not just specific platforms
  4. Cost: No massive inference costs or API dependencies
  5. Control: You decide when and how to upgrade each component

When Magic Breaks

Real Examples:

  • OpenAI API goes down → Your local ollama keeps working
  • New Claude model changes behavior → Your saved prompts work the same way
  • GitHub Copilot suggests garbage → Your manual scripting skills haven't atrophied
  • Vector database service gets expensive → You own your embeddings and can move them

The Beautiful Irony

Using "Advanced AI" the Primitive Way

  • Local computation
  • File-based storage
  • Shell scripting
  • Human oversight
  • Manual fallbacks

Primitive Systems Disguised as "Advanced"

  • Cloud dependencies
  • API subscriptions
  • Black box algorithms
  • Automated everything
  • No fallbacks

Core Tenets

  1. Use compute, not subscriptions - AI is code and data, not a service
  2. Component replaceability over monoliths - Everything breaks, make it fixable
  3. Local computation over cloud dependencies - Own your tools
  4. Human judgment in control loops - Automate execution, not decisions
  5. Boring technology + AI assistance - Reliable infrastructure with smart helpers
  6. The Unix philosophy for AI - Do one thing well, compose through standard interfaces

The Maintenance Reality

In 2 Years:

  • Your steampunk tools will still work (maybe need minor updates)
  • Magic AI systems will have broken 3 times and been "reimagined" twice
  • You'll understand your system better (more experience with components)
  • They'll understand their system less (more abstractions, more magic)

Software winter survivors will be the people who treated AI like a tool you own instead of a service you rent.
